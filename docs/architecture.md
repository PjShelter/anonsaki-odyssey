# Architecture

`anonsaki-odyssey` 的目标不是做一个大而全的通用引擎，而是做一个 **面向 2D 横版射击 + 轻量剧情展示 + 局外商店** 的项目结构。

一句话概括：

> **代码负责规则与运行时，内容负责角色、地图、武器、剧情与商店数据。**

---

## 1. 总体分层

项目建议分成 4 层：

1. **App / Scene 层**
   - Phaser 场景切换
   - 启动流程
   - UI 展示
2. **Game Runtime 层**
   - 战斗规则
   - 实体更新
   - 碰撞、伤害、状态
3. **Content 层**
   - 角色配置
   - 武器配置
   - 地图配置
   - 剧情脚本
   - 商店商品定义
4. **Persistence 层**
   - 本地存档
   - 设置项
   - 已解锁内容
   - 进度记录

这样拆的好处很直接：
- 场景不用直接写死具体角色数据
- 战斗逻辑和剧情内容能分开维护
- 后续做模组时，主要扩展的是内容层，不是硬改运行时代码

---

## 2. 推荐目录职责

```txt
src/
├─ main.ts                 # 启动入口，创建 Phaser.Game
├─ game/
│  ├─ scenes/              # Phaser 场景
│  ├─ engine/              # 运行时核心逻辑
│  ├─ entities/            # 玩家、敌人、子弹等实体
│  ├─ rigs/                # 角色/敌人行为组装
│  ├─ ui/                  # UI 组件
│  ├─ content/             # 内容加载与校验
│  ├─ save/                # 存档读写
│  └─ types/               # 共享类型定义

public/
├─ content/
│  └─ base/                # 官方基础内容包
└─ mods/                   # 模组内容
```

### scenes/
只负责“这一幕发生什么”。

比如：
- `BootScene`：预加载最低限度资源
- `MainMenuScene`：主菜单
- `DialogueScene`：剧情对话
- `BattleScene`：战斗主场景
- `ShopScene`：局外商店

场景层应该少做业务判断，别把数值和剧情规则全糊进去，不然后面会变成巨型屎球。

### engine/
放核心系统：
- 战斗循环
- 伤害结算
- 波次/刷怪控制
- 关卡状态
- 事件总线

这一层是“规则层”，尽量不依赖具体角色名字和具体地图文件。

### entities/
放运行时对象：
- `Player`
- `Enemy`
- `Projectile`
- `Pickup`

实体负责自己的状态、移动和基础行为，但不要自己偷偷决定全局规则。

### rigs/
这里适合放“装配后的玩法模板”。

比如：
- 某个角色默认武器配置
- 某类敌人的攻击模式
- Boss 行为阶段组合

它像预制件，不是底层引擎，也不是纯内容 JSON。

### ui/
放独立 UI 组件：
- 血条
- 弹药 UI
- 对话框
- 角色立绘槽位
- 商店面板

目标是让场景只负责组装 UI，不负责手搓每个文本和按钮状态。

### content/
负责：
- 加载 JSON 内容
- 合并 base + mods
- 校验字段
- 建立索引

这一层非常关键。以后你想支持模组，核心就是这里别写烂。

### save/
负责：
- 本地存档结构版本
- 存档迁移
- localStorage / IndexedDB 封装
- 设置、进度、商店解锁状态

不要让每个场景直接读写 localStorage，脏得很。

## 3. 运行流程

推荐启动流程：

```txt
main.ts
  -> BootScene
  -> MainMenuScene
      -> DialogueScene / BattleScene / ShopScene
```

更细一点：

1. `main.ts` 创建 Phaser 游戏实例
2. `BootScene` 初始化基础资源和内容注册器
3. 内容加载器扫描 `public/content/base`
4. 读取本地存档
5. 进入 `MainMenuScene`
6. 玩家从主菜单进入剧情、战斗或商店

如果以后加模组，模组加载发生在 Boot 或专门的 Content Bootstrap 阶段最合理。

---

## 4. 战斗架构

战斗部分建议采用 **数据驱动 + 小型运行时系统**，而不是一堆 if/else 把自己埋了。

### 核心对象

- **RunState**：当前战斗运行状态
- **PlayerState**：玩家属性、武器、Buff
- **EnemyState**：敌人模板实例化后的状态
- **StageState**：当前地图、波次、目标
- **CombatSystem**：驱动更新流程

### 推荐更新顺序

每帧或固定时间步大概这样：

1. 处理输入
2. 更新玩家
3. 更新敌人
4. 更新投射物
5. 处理碰撞
6. 结算伤害与死亡
7. 处理掉落与事件
8. 更新 UI
9. 判断关卡完成/失败

重点不是“高级”，重点是顺序固定。顺序一乱，bug 会像蟑螂一样满地跑。

---

## 5. 剧情系统架构

这个项目的剧情层要 **故意做小**。

目标能力就这些：
- 左 / 中 / 右 三个立绘槽位
- 当前说话角色高亮
- 角色表情切换
- 背景切换
- 对话文本推进
- 简单分支选项

### 推荐结构

- `DialogueScene`：专门负责剧情 UI
- `DialogueRunner`：顺序执行剧情节点
- `DialogueAssetResolver`：根据角色/表情找到立绘资源

### 剧情节点类型示例

- `background`
- `showPortrait`
- `hidePortrait`
- `say`
- `choice`
- `jump`
- `setFlag`

重点是让剧情写成数据，而不是把剧情写进 TypeScript 里。不然你后面改一句台词都要碰代码，纯折磨。

---

## 6. 商店架构

只做 **局外商店**，别把系统搞成网游后台。

商店负责：
- 解锁武器
- 永久属性升级
- 特殊被动解锁
- 剧情/进度门槛检查

### 商店建议拆分

- `ShopScene`：界面和交互
- `ShopService`：购买逻辑、合法性校验
- `ProgressionState`：玩家长期成长数据

### 关键原则

- 价格写在内容层
- 购买校验写在逻辑层
- 已购买结果写到存档层

别在按钮点击回调里又扣钱又写存档又判解锁条件，那种代码以后看一眼都想删库跑路。

---

## 7. 内容与模组架构

这是仓库后面能不能扩展的关键。

### 设计原则

- 官方内容放 `public/content/base`
- 模组内容放 `public/mods/<mod-id>`
- 所有内容通过统一加载器进入运行时
- 运行时代码不关心内容来自 base 还是 mod

### 内容加载步骤

1. 读取 base 内容清单
2. 读取 mod 清单（如果启用）
3. 按类型加载 JSON
4. 做 schema 校验
5. 建立 ID 索引
6. 冲突处理（覆盖 / 禁止重复 / 显式 patch）

### 内容覆盖策略建议

最稳妥的是：
- 默认不允许静默重名覆盖
- 模组若要覆盖，必须显式声明 patch 或 override

不然以后两个 mod 都叫 `weapon.pistol`，你会被自己坑死。

---

## 8. 存档设计

存档至少应该分成三块：

```txt
SaveData
├─ meta
├─ settings
├─ progression
└─ story
```

### meta
- 存档版本
- 创建时间
- 更新时间

### settings
- 音量
- 分辨率 / 显示设置
- 键位

### progression
- 金币 / 资源
- 武器解锁
- 永久升级
- 已通关关卡

### story
- 已读章节
- 分支标记
- 当前剧情进度

### 为什么要带版本号
因为你后面一定会改结构。没有版本迁移，老存档一炸，玩家就会想骂人。

---

## 9. 事件与通信

推荐在运行时使用轻量事件总线处理跨系统通知，例如：

- `battle:start`
- `enemy:killed`
- `player:damaged`
- `stage:cleared`
- `shop:purchase-completed`
- `story:flag-changed`

用途：
- UI 跟随状态变化
- 音效响应事件
- 成就或统计监听

但别滥用。不是所有函数调用都要绕一圈事件，那会把调试体验变成地狱。

---

## 10. 不做的东西

为了保持项目收敛，当前阶段明确 **不做**：

- 通用 galgame 引擎能力
- 在线多人同步
- 云端经济系统
- 服务端权威战斗判定
- 复杂任务树编辑器
- 可视化剧情编辑器

先把“能跑、能扩、结构不烂”做好，比空想一整套宏伟平台强一百倍。

---

## 11. 第一阶段建议落地顺序

建议按这个顺序推进：

1. 定义共享类型与内容 schema
2. 实现内容加载器
3. 搭 `MainMenuScene`
4. 搭 `DialogueScene` 三槽位 UI
5. 搭 `BattleScene` 空战斗循环
6. 搭 `ShopScene` 骨架
7. 接入本地存档

这顺序的好处是：
- 先把骨架钉死
- 再往里面灌内容
- 不会一开始就陷入具体玩法细节泥潭

---

## 12. 一句话总结

这个项目最合理的架构不是“大而全”，而是：

> **一个小型、清晰、可扩展的 Phaser 游戏运行时 + 一套数据驱动内容包结构。**

把边界守住，后面就舒服；边界一开始就糊，后面全是债。
